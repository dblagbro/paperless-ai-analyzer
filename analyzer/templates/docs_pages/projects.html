Projects

Organize documents into isolated workspaces — each project maintains its own AI vector store for scoped search and chat.

What is a project? A project is a named workspace (e.g., "Robinhood Fonda Case", "Client ABC Matter") that groups documents via a Paperless tag (project:slug). AI Chat and Search operate within the selected project only, ensuring answers never mix documents from different matters.

How projects work

When a document is uploaded with a project selected, or assigned to a project later, the analyzer adds a Paperless tag project:slug to the document. The analysis pipeline then embeds the document into that project's dedicated ChromaDB collection (paperless_docs_{slug}) rather than the default collection.

  • Isolated vector store — each project gets its own ChromaDB collection. Chat and search never see documents from other projects.
  • Paperless tag synced — the project:slug tag is visible in Paperless-ngx so documents can be found by project even outside the analyzer.
  • Default project — documents without a project tag go into Default Project. This cannot be deleted.

Creating a project

1

Open Manage Projects

Click the Manage Projects tab in the dashboard.

2

Click + New Project

The creation modal opens. Fill in:

  • Project Name — display name (e.g., "Robinhood Fonda v. Fairbridge")
  • Slug — auto-generated from the name. Must be lowercase alphanumeric with dashes/underscores, 2–50 characters. Once set, cannot be changed.
  • Description — optional free-text description.
  • Color — used for the project tag badge in Paperless and UI accents.
3

Save

The project is created in the local database and the corresponding project:{slug} tag is created in Paperless-ngx immediately.

Switching the current project

The Current Project dropdown in the top header controls the scope of AI Chat and Search. Select any project to switch. The change takes effect immediately — no page reload required.

  • AI Chat sessions created while a project is active are associated with that project's collection.
  • Search results only show documents from the current project's vector store.
  • The Overview tab always shows stats across all documents regardless of the current project.

Assigning documents to a project

Documents can be assigned at upload time (preferred) or after the fact:

  • At upload — select the project in the Smart Upload tab's Project dropdown before uploading. The project:slug tag is applied immediately.
  • After upload — in Manage Projects, click Assign Documents on a project card. Search for documents by title or tag and assign them individually or in bulk.

Migrating documents between projects

Click the 📦 Move button on a project card to open the Move Documents dialog. Select the source and destination projects, choose whether to move all documents or specify individual document IDs, then click Move Documents.

The migration runs in the background and:

  • Moves each document's ChromaDB vector embedding to the destination collection immediately (no re-embed needed).
  • Updates all processed_documents records to the new project slug.
  • Migrates all AI Chat sessions (and their messages) from the source project to the destination.
  • Updates the Paperless project:old-slug tag → project:new-slug on each document in Paperless.
  • Refreshes the cached document count on both projects when done.
Moving documents is logged: look for Migration completed: N docs moved to <slug>, E errors in the analyzer log. Any document that fails its Paperless tag update is counted as an error but does not block the rest of the migration.

Archiving vs. deleting

ActionEffectReversible?
ArchiveProject hidden from the active list. Documents and vector store preserved. Tag stays in Paperless.Yes — Unarchive restores it.

The Default Project cannot be deleted. It serves as the catch-all for documents without a project tag and ensures backward compatibility.

Orphan documents

Documents in Paperless that have no project: tag are called orphans. Click View Orphan Documents in the Manage Projects tab to see them and bulk-assign them to a project.

Project statistics

Each project card on the Manage Projects tab shows:

  • Documents — live count from the ChromaDB collection (always current).
  • Created / Last Analyzed — timestamps from the project database.
  • Storage — approximate disk size of the vector store.

Per-Project Paperless-ngx Instances (v3.6.1)

By default, all projects share a single Paperless-ngx instance with project:{slug} tags for separation. v3.6.0 added support for dedicated instances; v3.6.1 makes provisioning fully automated — no manual Docker, nginx, or SQL steps required.

Auto-Provisioning (v3.6.1, Recommended)

When a new project is created, provisioning starts automatically in the background. You can also trigger it manually for any existing project:

  1. Click ⚙️ Paperless on the project card to open the Configure Paperless modal.
  2. Click ⚡ Auto-Provision Now in the Provision tab.
  3. A live progress banner shows each step. Provisioning typically takes 1–3 minutes.
  4. On completion, the banner shows "✅ Provisioning complete" with an Open Paperless → link. The project is immediately wired to its new instance.

The provisioner performs all these steps automatically:

  • Reads shared config (DB password, timezone) from the running paperless-web container via Docker SDK
  • Assigns the next free Redis DB index (scans running paperless-web-* containers)
  • Generates a strong random secret_key and admin_password
  • Creates the per-project Postgres database on paperless-postgres (idempotent)
  • Creates host directories under /mnt/s/documents/paperless-{slug}/
  • Starts paperless-web-{slug} and paperless-consumer-{slug} containers with unless-stopped restart
  • Waits for the web container to become ready (up to 3 minutes)
  • Obtains an API token and writes the nginx location block to projects-locations.d/paperless-{slug}.conf
  • Reloads nginx and saves all credentials (AES-256-GCM encrypted) to the project DB
Prerequisites: The nginx service must mount ./config/nginx/projects-locations.d:/etc/nginx/projects-locations.d:rw and the analyzer container must mount the same directory at /app/nginx-projects-locations.d:rw. The nginx.conf 443 server block must include include /etc/nginx/projects-locations.d/*.conf;. These are one-time host setup steps (already done on this server as of v3.6.1).

Manual Provisioning (Advanced)

If you need to provision manually (e.g., on a different host), click ⚙️ Paperless → Provision → Manual Setup Snippets to expand the compose and nginx snippets, then:

  1. Copy the Docker Compose snippet into your docker-compose.yml.
  2. Copy the nginx block into your server's nginx config.
  3. Run: sudo docker compose up -d paperless-web-{slug} paperless-consumer-{slug}
  4. Open the public URL, create the admin user, and generate an API token from Admin → Auth Tokens.
  5. Switch to the Connect tab and enter the URL + token manually.

Connect tab

  1. Switch to the Connect tab in the Configure Paperless modal.
  2. Enter the Internal Container URL — the Docker-internal address, e.g. http://paperless-web-robinhoodproperties:8000.
  3. Enter the API Token from the new instance.
  4. Enter the Public Base URL — the nginx-proxied public URL, e.g. https://home.example.com/paperless-robin. This is used for "View in Paperless" links throughout the UI.
  5. Click Test Connection to verify, then Save.

From this point on, the project's Paperless API calls (analyze-missing, delete, upload, poll) all route to the dedicated instance. A dedicated polling thread also starts automatically on the next container restart.

Step 3 — Migrate existing documents

If you previously used the shared Paperless instance for this project, use the Migrate tab to move all existing documents:

  1. Click 📦 Start Migration.
  2. The migration downloads each document from the shared instance, uploads it to the new instance, waits for OCR to complete, then:
    • Re-keys ChromaDB embeddings to the new document ID.
    • Updates processed_documents in SQLite.
    • Patches chat_messages that reference old document IDs.
    • Updates court_imported_docs entries.
  3. A live progress bar shows N / total documents migrated.
  4. After migration completes, the old project:{slug} tags in the shared instance are no longer used as a data source for that project.
Note: The migration does not delete documents from the shared instance. You can remove them manually after verifying the migration was successful.

View in Paperless links

Once a Public Base URL is configured for a project, the UI shows clickable links throughout:

  • Manage Projects → Docs table — an ↗ View button next to each document.
  • Search / Analysis tab — document IDs become clickable links.
  • AI Chat — [Document #NNN] and Document #NNN references in assistant responses are automatically linkified.
  • Case Intelligence — evidence card document titles become links.
